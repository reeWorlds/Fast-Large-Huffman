Bucketed canonical huffman idea.

Lets say we have canonical huffman codes. The decoding algorithm has while loop "while (blockCode >= t_limit[l]) { l++; }". It is required to determine code length when "int32_t l = t_first[bitStream >> lookupShift];" fails.

The idea is to remove this while loop by slightly sacrificing compression ratio. If codes with prefix `p` have diffrent lengths (and these lengths >= k) => all codes with prefix `p` are forced to have the same length. This modified code is called "bucketed" because each bucket (corresponding to `p`) has fixed code length. The enforcement is implemented by just forcing all codes to max length in bucked.

Example.
Lets say k = 2 and we have:
codes 0, 100, 101, 110,  1110, 1111 with lengths 1, 3, 3, 3, 4, 4.
We can see that bucket `p=11` has lengths 3 and 4 (e.g. "11|0" vs "11|10"). Bucketed code could be:
codes 0, 100, 101, 1100, 1101, 1110 with lengths 1, 3, 3, 4, 4, 4.

For implementation look at lines 115-134 of HuffV6_1_1.cpp. Some constant defintions are moved up (lines 11-13), but everything else is identical to Opt_1_3_4.cpp.
Note that `k` (LOOKUP_BITS) should be identical for compressor and decompressor.

Why this near-optimal?
For typical `k = 10` or `k = 12` there are 1024 or 4096 buckets. Out of them only buckets with different lengths are adjusted. The trick is that only buckets crossing code-length boundaries need adjustment, so the number of adjusted buckets is bounded by about `max_code_len - k`. Even on large text with `max_code_len = 32` (and considering `k = 12`) there would be about 20 out of 4096 buckets with adjusted codes. Not all codes in these buckets have their length increased (only ones with smaller initial length). Also, code lengths are typically increased by 1 bit which is small relative to original code length (e.g. 15->16 means +6.6%).
In conclusion: only small number of buckets (20 out of 4096) are adjusted, not all codes in these 20 buckeds are lengthened, and even lengthened codes have small overhead (2-8%).
